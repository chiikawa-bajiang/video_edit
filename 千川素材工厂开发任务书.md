# 千川素材工厂开发任务书

版本：V1.0  
面向对象：开发负责人、Cursor/Codex 多 Agent、运营协作人员  
最终目标：交付一个可在 Windows 上执行的本地 `exe` 项目，用于服装类千川素材的批量生产、管理、质检、导出和数据复盘。

---

## 1. 项目定位

本项目不是单纯的视频剪辑工具，也不是单点 AI 生成工具，而是一个“千川素材生产工厂”。

核心目标：

- 输入：每天约 20 条服装原始视频。
- 处理：自动拆分、标记、组合、包装、渲染、质检。
- 输出：每天 500-2000 条可用于千川测试的素材。
- 复盘：回收投放数据，沉淀高 ROI 的素材结构。
- 交付：最终形成一个 Windows 可执行程序，运营可以在本地使用。

项目不强制限定语言和框架，但建议优先使用 `npm` 生态完成前中期，因为：

- 适合 Cursor/Codex 多 Agent 分工开发。
- 能同时处理桌面 UI、任务队列、FFmpeg 调用、文件管理、配置管理。
- 后续可通过 Electron 或 Tauri 打包为 `exe`。
- 前期可以先做 CLI，再升级为桌面应用。

## 2. 推荐技术路线

### 2.1 推荐方案

优先推荐：

```text
TypeScript + Node.js + FFmpeg + SQLite + Electron/Tauri + 本地文件系统
```

建议阶段化落地：

| 阶段 | 形态 | 技术建议 | 目标 |
|---|---|---|---|
| 前期 | CLI + 本地目录 | Node.js / TypeScript / FFmpeg | 跑通素材入库、模板组合、批量渲染 |
| 中期 | 本地管理后台或桌面端 | Electron 或 Tauri + SQLite | 支持运营可视化操作、任务管理、数据回流 |
| 后期 | 完整 exe 工具 | Electron Builder / Tauri Build | 一键安装、配置、批量生产、复盘推荐 |

### 2.2 可替代方案

如果团队更熟悉其他技术，也可替换：

| 方向 | 可选技术 |
|---|---|
| 后端脚本 | Python / Go / Rust / Node.js |
| 桌面端 | Electron / Tauri / Qt / Wails |
| 数据库 | SQLite / DuckDB / PostgreSQL |
| 队列 | 本地任务队列 / Redis / BullMQ / SQLite job table |
| 视频处理 | FFmpeg 必选，其他工具作为辅助 |
| AI 能力 | OpenAI / 通义 / Gemini / 本地模型 / 可插拔 API |

前中期建议不要过早上复杂服务端架构，先保证本地可运行、可交付、可验证。

## 3. 最终产品形态

最终交付物建议是一个 Windows 桌面 `exe`：

```text
QianchuanMaterialFactory.exe
```

运营打开后可以完成：

1. 导入原始视频。
2. 绑定 SKU、价格、卖点、库存、模特、背景。
3. 自动切片或手动标记镜头段。
4. 选择生产模板。
5. 设置生成数量。
6. 批量渲染素材。
7. 自动生成素材命名和导出目录。
8. 质检通过后导出给千川投放。
9. 导入投放数据。
10. 查看 ROI、CTR、CVR、素材生命周期和模板效果。

## 7. 多 Agent 协同开发方案

### 7.1 Agent 分工原则

多 Agent 不要同时改同一个文件。每个 Agent 负责一个边界清晰的模块，并交付文档、代码、测试和接口说明。

建议分工：

| Agent | 职责 | 主要目录 |
|---|---|---|
| Agent-01 架构负责人 | 项目脚手架、模块边界、统一类型、开发规范 | 根目录 / packages/core |
| Agent-02 素材资产 | 原片导入、SKU 管理、文件扫描、元数据读取 | packages/core/src/asset |
| Agent-03 视频切片 | FFmpeg 探测、切片、镜头段管理 | packages/core/src/segment |
| Agent-04 模板组合 | 组合规则、素材矩阵、版本生成 | packages/core/src/template |
| Agent-05 渲染中心 | FFmpeg 命令生成、批量渲染、失败重试 | packages/ffmpeg |
| Agent-06 质检导出 | 分辨率、时长、黑屏、字幕、命名、导出目录 | packages/core/src/quality 和 export |
| Agent-07 数据回流 | 千川数据导入、指标计算、ROI 模型 | packages/core/src/metrics |
| Agent-08 桌面端 | Electron/Tauri UI、任务进度、配置页面 | apps/desktop |
| Agent-09 打包发布 | exe 打包、安装包、版本升级、运行环境检查 | build / release |
| Agent-10 测试验收 | 集成测试、样例数据、冒烟测试、验收清单 | tests / docs |

### 7.2 多 Agent 协作规则

每个 Agent 开始前必须明确：

- 本次只改哪些目录。
- 输入数据是什么。
- 输出接口是什么。
- 是否会影响其他模块。
- 如何运行测试。

每个 Agent 完成后必须交付：

- 变更摘要。
- 使用方式。
- 测试命令。
- 已知限制。
- 给下游 Agent 的接口说明。

### 7.3 推荐任务分发顺序

```text
第 1 批并行：
Agent-01 架构负责人
Agent-02 素材资产
Agent-05 渲染中心

第 2 批并行：
Agent-03 视频切片
Agent-04 模板组合
Agent-06 质检导出

第 3 批并行：
Agent-07 数据回流
Agent-08 桌面端
Agent-10 测试验收

第 4 批：
Agent-09 打包发布
```

## 8. 前期阶段：0-30天

前期目标：不要追求完整系统，先做一个可运行、可批量生成素材的 MVP。

### 8.1 前期业务目标

- 能导入原始视频。
- 能配置 SKU 和卖点。
- 能把视频拆成镜头段。
- 能通过模板组合生成素材。
- 能调用 FFmpeg 批量渲染。
- 能导出统一命名的视频文件。
- 能让运营每天实际使用。

### 8.2 前期功能范围

#### P0 功能

| 功能 | 说明 | 验收标准 |
|---|---|---|
| 项目脚手架 | npm monorepo、TypeScript、基础命令 | `npm install` 和 `npm run build` 可成功 |
| 工作目录初始化 | 创建 input/output/temp/segments/reports | 首次运行自动创建目录 |
| 原片导入 | 扫描指定目录视频文件 | 能读取 mp4/mov 基础信息 |
| SKU 配置 | 支持 JSON/CSV 导入 SKU | 能把原片绑定到 SKU |
| 手动切片配置 | 通过 JSON 指定 start/end/action | 能按配置切出片段 |
| 模板组合 | 支持基础组合规则 | 能生成待渲染任务列表 |
| FFmpeg 渲染 | 拼接、裁切、变速、字幕、配乐 | 能批量导出视频 |
| 命名规则 | 自动生成素材 ID | 文件名可追踪 SKU、模板、版本 |
| 运行日志 | 记录成功和失败 | 失败任务可定位原因 |

#### P1 功能

| 功能 | 说明 | 验收标准 |
|---|---|---|
| 自动切片辅助 | 基于固定时间段或简单规则切片 | 减少人工配置成本 |
| 简单质检 | 时长、分辨率、文件大小、是否生成成功 | 不合格素材进入 rejected |
| 批量任务队列 | 并发控制、失败重试 | 大批量渲染不崩溃 |
| 模板配置文件 | 用 JSON/YAML 管理模板 | 运营可复制模板修改 |
| 生产报告 | 输出每日生产数量、失败数量 | 生成 markdown/csv 报告 |

### 8.3 前期推荐 CLI 命令

```bash
npm run init-workspace

npm run segment -- --asset asset_001 --config ./templates/segments/default.json
npm run plan -- --sku SKU001 --template basic_matrix --count 100
npm run render -- --plan ./workspace/temp/render-plan.json
npm run report -- --date today
```


### 8.5 前期验收标准

前期完成时，需要达到：

- 单机能完整跑通从原片到成片的流程。
- 20条原视频可以稳定生成至少500条素材。
- 每条素材有唯一 ID。
- 每条素材能追溯到 SKU、模板、原片和切片。
- 渲染失败有日志，不影响整体任务继续。
- 运营可以按照文档操作，不需要开发全程陪同。

## 11. exe 打包方案

### 11.1 Electron 方案

适合：

- 团队熟悉 npm。
- 希望 UI 开发快。
- 可以接受包体较大。

推荐依赖：

```bash
npm install electron electron-builder
```

推荐脚本：

```json
{
  "scripts": {
    "desktop:dev": "electron .",
    "desktop:build": "electron-builder --win",
    "build:exe": "npm run build && npm run desktop:build"
  }
}
```

输出：

```text
release/
  QianchuanMaterialFactory Setup.exe
  win-unpacked/
```

注意：

- FFmpeg 需要随安装包一起分发，或首次启动时检查本机是否存在。
- 本地数据库建议放在用户数据目录，不要放在程序安装目录。
- 工作目录允许用户自行选择，避免 C 盘权限问题。

### 11.2 Tauri 方案

适合：

- 希望包体更小。
- 可以接受 Rust 环境。
- 项目后期更重视性能和系统集成。

缺点：

- 前期配置成本比 Electron 高。
- 多 Agent 需要有人熟悉 Rust/Tauri。

### 11.3 推荐选择

前中期建议：

```text
CLI MVP -> Electron 桌面端 -> electron-builder 打包 exe
```

原因：

- 与 npm 生态一致。
- Cursor/Codex 更容易并行开发。
- UI、文件管理、FFmpeg 调用都比较直接。
- 对运营交付更快。

## 12. 关键技术点

### 12.1 FFmpeg 能力

前期必须支持：

- 视频信息读取。
- 指定时间切片。
- 视频拼接。
- 画面裁切。
- 比例转换。
- 变速。
- 字幕叠加。
- 背景音乐叠加。
- 导出 mp4。

中期再支持：

- 封面图导出。
- 黑屏检测。
- 音量标准化。
- 多分辨率预设。
- 渲染并发控制。

### 12.2 命名规则

建议素材 ID：

```text
SKU001_TPL03_HOOK02_SP05_V012_20260617.mp4
```

含义：

- `SKU001`：商品。
- `TPL03`：模板。
- `HOOK02`：开头类型。
- `SP05`：卖点。
- `V012`：版本号。
- `20260617`：生成日期。

命名必须稳定，否则投放数据无法回流。

### 12.3 配置文件示例

模板配置：

```json
{
  "id": "TPL_BASIC_001",
  "name": "展示-走路-转身-促销",
  "durationTarget": 12,
  "segments": ["show", "walk", "turn", "ending"],
  "captionStyle": "white-bold-bottom",
  "speed": [0.95, 1.0, 1.08],
  "aspectRatio": "9:16",
  "music": ["music_001.mp3", "music_002.mp3"]
}
```

生产计划：

```json
{
  "skuId": "SKU001",
  "templateId": "TPL_BASIC_001",
  "count": 100,
  "hooks": ["显瘦10斤", "梨形身材必看", "通勤这样穿"],
  "sellingPoints": ["遮胯", "显腰线", "不挑身材"]
}
```

## 14. 项目风险

| 风险 | 影响 | 应对 |
|---|---|---|
| 一开始功能做太多 | MVP延期 | 前期只做生产闭环 |
| 高级 AI 成本过高 | ROI不稳定 | 只对 TOP 素材使用 |
| 命名不统一 | 数据无法回流 | 第一阶段就固定素材 ID |
| FFmpeg 参数失控 | 质量不稳定 | 参数模板化 |
| 运营不会用 | 工具闲置 | 前期 CLI 后必须尽快做 UI |
| 只追求2000条数量 | 投放浪费 | 必须同时做数据复盘 |

## 15. 需要你确认的细节

以下问题会影响开发优先级，但不影响先按本文档开工：

1. 原始视频主要格式是 `mp4` 还是也有 `mov`？
2. 每条原始视频平均时长是多少？
3. 素材最终主要输出比例是 `9:16` 吗？
4. 是否需要自动加字幕，还是只需要视频画面变体？
5. 字幕文案由运营提供，还是需要 AI 自动生成？
6. 千川数据能否导出 CSV/Excel？里面是否能包含素材文件名或素材 ID？
7. exe 是只给内部团队使用，还是要给外部客户安装？
8. 是否需要登录、权限和多人协作？
9. 电脑配置大概是什么水平，是否有独立显卡？
10. FFmpeg 是否允许随 exe 一起打包分发？

## 18. 结论

本项目建议先用 npm 生态完成 MVP，再升级桌面端，最后打包 exe。

前期最重要的是：

- 统一素材 ID。
- 跑通原片到成片。
- 支持批量渲染。
- 输出可投放素材。

中期最重要的是：

- 桌面端可视化。
- 任务队列稳定。
- 千川数据回流。
- ROI 看板。
- 模板沉淀。

后期再考虑：

- 自动推荐。
- 高级 AI 强化。
- 多账号协作。
- 云端同步。

只要前中期做好，这个项目就已经可以作为一个可执行的千川素材生产工具交付给运营使用。
